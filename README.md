# spastic-typer

A personality assessment and comparison tool covering three systems: **Enneagram**, **MBTI**, and **Instinct Stack**. Users take adaptive quizzes, receive detailed typed profiles, and can analyze the dynamics between 2–6 people on the Compare page.

**Live app:** [lesps.github.io/spastic-typer](https://lesps.github.io/spastic-typer/)

---

## Features

### Guided Typer

Three independent adaptive quizzes, each using a confidence-based early-exit algorithm that stops asking when the result is statistically clear:

- **Enneagram** — Scores all 9 types using a 45-question bank (5 per type), delivered in a fair interleaved sequence. When the top two types remain within threshold after the bank is exhausted, a disambiguation branch asks targeted clarifying questions. Returns core type, wing, wing strength (None / Slight / Moderate / Strong), and instinct stack (e.g. `4w5 Moderate SX/SP/SO`).
- **MBTI** — Scores all four dimensions (EI / SN / TF / JP) using a 32-question bank (8 per dimension) on a −3 to +3 Likert scale. Returns a 4-letter type with the full 8-function cognitive stack.
- **Instinct Stack** — Scores the three instinctual drives (SP / SX / SO) using a dedicated 15-question bank. Returns a ranked drive stack independent of the Enneagram result.

Completing all three quizzes unlocks the **Share Link** (encodes the full profile into a URL hash) and the **Export** button.

**Archetype name:** When all three systems are complete, a unique three-word archetype name is derived from the combination — for example, `INFP + Type 4 + SX-dominant` → *"Lyrical Introspective Seeker"*.

---

### Compare

Pairwise and group dynamics analysis for 2–6 people. Load profiles via any combination of:

- **Share URL** — paste a link generated by the Typer
- **JSON upload** — load a `.json` backup exported from the Typer
- **Manual entry** — pick Enneagram type, wing, wing strength, instinct stack order, and MBTI type from dropdowns

For each pair the Compare page shows:
- Enneagram interaction dynamics (from pre-computed lookup)
- Wing dynamics
- Instinct stack compatibility
- MBTI cognitive-function overlap and tips

When 3 or more profiles are loaded, a **Group Analysis** section summarizes patterns across the whole set.

---

### Explorer

A tabbed reference tool with four tabs:

| Tab | Contents |
|-----|----------|
| **Enneagram** | All 9 types — name, description, core fear/desire, growth/stress arrows, wings, instinct compatibility |
| **MBTI** | All 16 types — name, description, full 8-function cognitive stack |
| **Instinct** | SP / SX / SO — description, dominant/secondary/repressed dynamics, pairwise compatibility |
| **Integration** | Cross-system overview showing how the three systems relate |

Explorer deep-links from quiz result screens let users jump directly to the relevant type detail.

---

### Mental Model

A multi-tab reference for understanding the typing framework:

| Tab | Contents |
|-----|----------|
| **MBTI** | Quadrant map of all 16 types; tap any type for full detail |
| **Enneagram** | Circle diagram; tap any type for detail |
| **Instinct Stack** | Detailed breakdown of each drive's dominant/secondary/repressed expression |
| **Combined** | Cross-system archetype view; reads your saved quiz results |

Also hosts the **4-Step Typing Methodology (SOP)** with common pitfalls for each system.

---

### Export

From the Typer choose screen (once all three quizzes are complete):

- **Share Link** — copy a URL that encodes the full profile as a hash fragment
- **Export JSON** — download a machine-readable backup of the raw quiz results
- **Export Markdown** — download a human-readable profile report with AI context notes (communication preferences, stress patterns, growth edges, tips for AI assistants)

## Architecture

### View Switching

There is no router. `App.jsx` holds a `view` state string; `BottomNav` calls `setView()` to navigate between four views:

```
App.jsx
  ├── {view === 'typer'}    → <GuidedTyper />
  ├── {view === 'explorer'} → <Explorer initialTab={explorerTab} />
  ├── {view === 'model'}    → <MentalModel />
  └── {view === 'compare'}  → <ComparePage />
```

### Quiz Data Flow (GuidedTyper)

```
User selects quiz (enn | mbti | inst)
  → buildFairSequence(bank, keyFn)   — interleaved shuffle, one question per category per round
  → user answers stored in answers{} map: { questionIndex: likertValue }
  → after each answer: confidence check
      → if confident: show result immediately (early exit)
      → if bank exhausted (Enneagram): run disambiguation branch if top-2 within threshold
      → otherwise: advance to next question
  → scoreXxx(answers, seq, ...) → result object
  → write result to localStorage
  → return to choose screen
```

### Compare Data Flow

```
Entry (URL hash | JSON upload | manual form)
  → parse → profiles[] array
  → for each pair:
      getEnnInteraction()            — ENN_DYNAMICS lookup in pairLookup.js
      getMBTIInteraction()           — MBTI_INSIGHTS lookup in pairLookup.js
      getInstinctStackInteraction()  — INSTINCT_STACK_DYNAMICS lookup in pairLookup.js
  → render dynamics cards per pair
  → if 3+ profiles: analyzeGroup()  — cross-set pattern summary
```

### Pre-computed Pair Data

`src/data/pairLookup.js` (~7,000 lines) contains all possible pairwise dynamics pre-computed at build time. It is **machine-generated** — do not hand-edit. To regenerate after changing dynamics logic:

```bash
node scripts/generatePairs.mjs
```

### Share URL Format

Profile data is encoded as a URL hash fragment so it can be loaded directly into the Compare page:

```
#p1=4w5:strong:sx/sp/so:INFP&p2=8w9:moderate:sp/so/sx:ENTJ
```

Fields per person: `{ennType}w{wing}:{wingStrength}:{instStack}:{mbtiType}`

## Development Process

### Test-Driven Development

Every change to quiz logic, scoring, or UI flow should be preceded by a failing test:

1. Write a failing test in the appropriate file (`scoring.test.js` for pure logic, `*.test.jsx` for UI)
2. Implement the feature or fix
3. Run `npm test` — all tests must pass
4. Commit

See `CLAUDE.md` for the full TDD guide, exported test helper reference, and writing conventions.

### Styling

All CSS is inline via JavaScript objects. No `.css` files, no CSS-in-JS libraries, no UI component libraries.

```js
// Correct
<div style={{ background: G.bg2, color: G.text, border: `1px solid ${G.goldBorder}` }}>

// Wrong
<div className="card">
```

Always use `G.*` color tokens from `styles/theme.js`. Never hardcode hex values. Reusable style patterns go in `styles/styles.js`; one-off local styles stay inline.

### localStorage Keys

| Key | Contents |
|-----|----------|
| `typer_enn` | `{ coreType, wing, wingStrengthDelta, instinctStack, display, scores }` |
| `typer_mbti` | `{ result, scores }` |
| `typer_inst` | `{ instinctStack, instScores }` |

Always parse with `JSON.parse(localStorage.getItem(key))` and guard for `null`.

### Adding a Feature — Checklist

1. Write a failing test describing the expected behavior
2. Implement the feature
3. Run `npm test` — all tests green
4. If you touched a question bank, verify data integrity tests still pass
5. If you added a new exported function from `GuidedTyper.jsx`, add a unit test in `scoring.test.js`
6. Commit with a clear message
7. Update `CHANGELOG.md` with the change under the current `[X.Y]` section
8. If the change affects the public interface, architecture, or conventions, update `README.md` and/or `CLAUDE.md`
